Maya: Precisely. If a server is stateful, meaning it stores session-specific information locally, like a user's login credentials or the contents of their shopping cart...

Maya: It does. To achieve true horizontal scalability and that flexibility, we have to force the application servers to be completely stateless. All that session data, user tokens, any temporary state, it must be moved out of the application server's memory and into a shared external resource.

Maya: Typically a fast, redundant cache like Redis, or maybe a dedicated session database. If a server is stateless, the LB can route your first request to server A, your second request to server B, and your third to server C, and you, the user, will never notice the difference.

Maya: The slave databases, which are often referred to as read replicas, are kept synchronized with the master, and they handle the vast majority of all the read operations.

Maya: Incredibly. It's often 10 to 1, and sometimes as high as 200 to 1. You read data far more often than you write it. So we can now offload that massive read load across multiple slave machines.

Maya: The failed slave is replaced, it gets resynced from the master, and it's added back into the pool. Data persistence is never threatened.

Maya: It is the most complicated architectural problem at this stage of scaling. When the master fails, the system has to execute a critical failover process. Engineers have to run sophisticated data recovery and reconciliation scripts to confirm the new master is the most up-to-date authority. Maya: Right. This means reducing those high-latency round trips to the database as much as humanly possible. Caching is the answer here.

Maya: It's really an elegant application of the locality of reference principle.

Maya: If a piece of data has been requested recently, it is highly probable it will be requested again soon. Ken: Correct. So let's look at the basic mechanism used to integrate this cache: the read-through strategy. Can you walk us through how the web server decides whether to use the cache or go all the way to the database?
Maya: Sure. When the application logic needs a piece of data, the web server initiates the process. Step one is always: check the cache.
Ken: Always check the cache first.
Maya: Always. If the data is present and it hasn't expired, we call that a cache hit. The data is instantly retrieved from memory and returned to the client. This bypasses the database entirely and saves that expensive disk IO cost.

Maya: If the data is not found, a cache miss, the application server then has to go and query the persistent database. Once the database returns the response, the web server performs two critical actions at the same time.

Maya: It stores a copy of that data in the cache, which primes the cache for the next request, and then it returns the data to the client. So the cost of that initial request is high, but all subsequent requests for that same data are drastically faster.

Maya: Perfect example. And you can never use the cache as the primary persistent store for important data. If a cache server restarts, all that data is gone. It is a performance layer, not a storage layer. And that leads directly to the problem of expiration.

Ken: What are they?
Maya: If the TTL is too short, say 30 seconds, the cache data expires so quickly that you're forcing frequent, unnecessary trips back to the database, which kind of defeats the purpose of the cache in the first place.
Ken: Right.
Maya: But if the TTL is too long, say 24 hours, you guarantee extremely high performance, but you run the risk of serving dangerously stale data to your users for a full day.
Ken: And consistency problems get exponentially worse when data spans multiple geographic regions. Our sources mentioned Facebook's work on scaling Memcached. Why is consistency so difficult to maintain when caches are distributed all over the world?

Ken: It's not one single step.
Maya: It's not. When data is written in data center A, that write must be asynchronously replicated to caches and databases in data centers B and C. There is an inherent window of inconsistency. Ken: We also can't forget that if we rely entirely on one cache server, we've just swapped one single point of failure (the master DB) for another one, the cache server.

Maya: Spreading the risk. Furthermore, engineers should always over-provision memory. Run the cache with perhaps 20% more memory than is strictly necessary.
